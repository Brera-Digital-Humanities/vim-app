# VIM — source & form reference

What the app is made of: source layout, screens, languages, the form data model
and conventions. For setup, build, run, deploy and troubleshooting see
[`../SETUP.md`](../SETUP.md).

## Source layout (`src/`)

The code is **vanilla JS/SCSS/HTML, no framework, no runtime dependencies**.
Files are concatenated (order in `build.order`) and inlined into single-file
pages by `scripts/build-app.sh`.

```
src/
├── build.order          Concatenation order: [html], [js], [scss]
├── app.html             PWA shell (body.app)  → dist/index.html
├── demo.html            Demo shell with phone skin (body.demo) → demo-desktop/index.html
├── partials/app-bar.html  App bar shared by both shells (via <!-- @file:… -->)
├── data.js              PAGES + CHOICES + RELEVANT + public Kobo UID (generated by `npm run sync`)
├── api.js               buildSubmissionXml() + doSubmit() → VIM backend
├── manifest.json        PWA manifest
├── assets/logo.svg      Brand logo (inlined as a data-URI at build time)
├── pwa/                 service-worker.js + icons/
├── i18n/                One language per file (it/en/ar) + index.js (UI_LANGS)
├── core/                Shared logic: state, storage, auth, router, relevant,
│                        labels, i18n-runtime, connectivity, feedback, init
└── screens/             One folder per screen (html + js + scss):
                         login · lang · home · form · drafts · outbox · sent · account
styles/                  Design system (tokens, layout, components, rtl, …)
```

Screens are inserted into the shells at the `<!-- @screens -->` marker; the app
bar at `<!-- @file:partials/app-bar.html -->`. In `build.order`: `[js]` puts
global `const`s first and `init.js` last; `[scss]` puts `tokens.scss` first.

`src/` is the **single source of truth** — never edit `dist/` or
`demo-desktop/` (generated). Build with `npm run build`.

## Languages

Italian, English, Arabic (RTL). Each is a `const VIM_LANG_xx = { key, name, rtl,
ui: {…} }` in `src/i18n/`. To add one:

1. Create `i18n/xx.js`, copy every `ui` key from `it.js` and translate it.
2. Add `VIM_LANG_xx` to the array in `i18n/index.js`.
3. Add `i18n/xx.js` to the `[js]` section of `build.order`.
4. Add the matching `label_xx` / `hint_xx` columns to the Kobo form (field
   labels come from there, via `npm run sync`).

## The form

Defined on **KoboToolbox** and pulled into `data.js` by `npm run sync` — never
edit `data.js` by hand. It is a grouped XLSForm: each `begin_group` becomes a
`PAGES` entry, each `select_one/_multiple` list a `CHOICES` entry. The app shows
**one field per screen**, grouped into sections, in Kobo's order.

**Completion** — the "Complete" button unlocks when every field that is both
*required* (Kobo `required`) and *currently visible* is filled, in any section
(`updateCompleteBtn()` in `screens/form/form.js`). Hidden conditional fields
don't block.

**Conditional fields** (`RELEVANT` in `data.js`, evaluated by
`core/relevant.js`) use XLSForm syntax, e.g. `${file_type}='tipo_file_19_1'`
with `and` / `or`. `isVisible(name)` → `evalRelevant(expr)` substitutes
`${field}` with `answers[field]` and evaluates the expression.

## Data & state (`core/state.js`)

Global variables for one session; drafts/outbox/sent are persisted in IndexedDB
(see `core/storage.js` and SETUP §"Offline").

| Variable | Meaning |
|---|---|
| `pageIdx`, `window._fieldIdx` | current section / field index |
| `answers` | `{ name: value }` (media fields hold the file name) |
| `mediaFiles` | `{ name: File }` |
| `drafts` | `[{ id, answers, mediaFiles, pageIdx, fieldIdx, savedAt, label }]` |
| `outbox` | `[{ id, label, answers, mediaFiles, xml, schemaSig, autoSend, savedAt }]` |
| `sentForms` | `[{ id, label, sentAt, answers }]` (text only, no media) |
| `currentLangIdx` | active language index in `UI_LANGS` |

`id` is a stable OpenRosa `instanceID` (`uuid:…`), kept from draft through
submission so re-sends are deduplicated server-side.

## Submission

`buildSubmissionXml(ans, mf, id)` builds the OpenRosa/XForm XML once at
completion (stored on the outbox record, so it survives later form changes).
`doSubmit(xml, mf)` POSTs it (multipart) to the **VIM backend**, which forwards
it to KoboToolbox with the server-side Kobo token. See SETUP §"Offline" for the
queue, auto-send and idempotency.

## Conventions

- Vanilla JS ES6+, no framework, no runtime deps.
- **CSS variables only** (`var(--accent)`, never raw hex).
- Shared list/card styles live in `styles/screens-base.scss` (`.list`,
  `.list-card`, `.card-*`) — don't inline repeated styles in JS.
- Modals are appended inside `.phone-shell`.
- Test RTL (Arabic) on every UI change.
- `npm run check` (syntax) and `npm test` (unit + jsdom) before committing.
